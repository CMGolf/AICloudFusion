## Concepts

Before we start, here is what each concept means:

**Amazon GuardDuty** is a continuous threat detection service that monitors your AWS account for malicious activity and unauthorized behavior. It analyzes CloudTrail logs, VPC Flow Logs, and DNS logs to identify threats. Think of it as a security guard watching camera feeds 24/7 — it never sleeps and alerts you when something looks wrong.

**Detector** is the GuardDuty resource that represents the service being enabled in a specific region. When you "enable GuardDuty," you are creating a detector. Each region has its own detector.

**Finding** is a security alert generated by GuardDuty when it detects suspicious activity. Each finding includes what happened, how severe it is, which resource is affected, and recommended remediation steps.

**Severity Levels** indicate how urgent a finding is:
- **Low (1.0–3.9):** Suspicious activity that may not be an immediate threat. Investigate when you have time.
- **Medium (4.0–6.9):** Activity that deviates from normal behavior. Investigate soon.
- **High (7.0–8.9):** A resource is compromised or under active attack. Investigate immediately.

**Sample Findings** are simulated threat findings that GuardDuty generates for testing purposes. They look exactly like real findings but are marked as samples. They do not indicate actual threats in your account.

### Step 5: Generate Sample Findings

Now you will generate sample findings that simulate real-world threats. These are safe — they do not indicate actual attacks on your account.

📋 Copy and paste, **replacing `<DETECTOR_ID>`**:

```
aws guardduty create-sample-findings --detector-id <DETECTOR_ID> --finding-types "UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom" "Recon:EC2/PortProbeUnprotectedPort" "CryptoCurrency:EC2/BitcoinTool.B!DNS" --region us-east-1
```

**✅ No output means success.** GuardDuty has generated three sample findings.

> **What do these finding types mean?**
>
> | Finding Type | What It Simulates |
> |-------------|-------------------|
> | `UnauthorizedAccess:IAMUser/MaliciousIPCaller.Custom` | An API call was made from a known malicious IP address |
> | `Recon:EC2/PortProbeUnprotectedPort` | An EC2 instance is being port-scanned by an attacker |
> | `CryptoCurrency:EC2/BitcoinTool.B!DNS` | An EC2 instance is communicating with a cryptocurrency mining pool (likely compromised) |

### Step 7: Get Finding Details

Now retrieve the full details of one finding to understand what GuardDuty reports.

📋 Copy and paste, **replacing `<DETECTOR_ID>` and `<FINDING_ID>`**:

```
aws guardduty get-findings --detector-id <DETECTOR_ID> --finding-ids <FINDING_ID> --region us-east-1 --query "Findings[0].{Type:Type,Severity:Severity,Title:Title,Description:Description}" --output table
```

**✅ You should see** a table showing the finding details:

```
-------------------------------------------------------------------
|                          GetFindings                             |
+-------------+---------------------------------------------------+
|  Description|  An API was invoked from an IP address...         |
|  Severity   |  5                                                |
|  Title      |  API invoked from a known malicious IP address.   |
|  Type       |  UnauthorizedAccess:IAMUser/MaliciousIPCaller...  |
+-------------+---------------------------------------------------+
```

> **💡 Understanding the output:**
> - **Type** — The category of threat (e.g., UnauthorizedAccess, Recon, CryptoCurrency)
> - **Severity** — A number from 0.1 to 8.9 indicating urgency
> - **Title** — A human-readable summary of what happened
> - **Description** — Detailed explanation of the finding

### Step 8: View All Findings with Severity

Let's see all findings at once with their severity levels.

📋 Copy and paste, **replacing `<DETECTOR_ID>`**:

```
aws guardduty list-findings --detector-id <DETECTOR_ID> --region us-east-1 --query "FindingIds" --output text
```

This gives you all finding IDs. Now get details for multiple findings:

📋 Copy and paste, **replacing `<DETECTOR_ID>`** (this retrieves details for up to 50 findings):

```
aws guardduty get-findings --detector-id <DETECTOR_ID> --finding-ids $(aws guardduty list-findings --detector-id <DETECTOR_ID> --region us-east-1 --query "FindingIds[:5]" --output text) --region us-east-1 --query "Findings[].{Type:Type,Severity:Severity,Title:Title}" --output table
```

> **💡 If the above command does not work on Windows**, use this two-step approach instead:

**Windows (PowerShell) alternative:**

📋 Copy and paste, **replacing `<DETECTOR_ID>`**:

```powershell
$findings = aws guardduty list-findings --detector-id <DETECTOR_ID> --region us-east-1 --query "FindingIds[:5]" --output text
aws guardduty get-findings --detector-id <DETECTOR_ID> --finding-ids $findings --region us-east-1 --query "Findings[].{Type:Type,Severity:Severity,Title:Title}" --output table
```

**✅ You should see** a table showing multiple findings with their severity levels.

### Step 10: Understanding Severity Levels

Here is how GuardDuty categorizes threat severity:

| Severity Range | Level | What It Means | Action Required |
|---------------|-------|---------------|-----------------|
| 7.0–8.9 | **High** | Resource is compromised or under active attack | Investigate immediately — stop the bleeding |
| 4.0–6.9 | **Medium** | Suspicious activity deviating from normal behavior | Investigate soon — could be an early indicator |
| 1.0–3.9 | **Low** | Suspicious but low-risk activity | Review when possible — may be a false positive |

**Real-world examples of each level:**

| Level | Example Finding | What Happened |
|-------|----------------|---------------|
| High | `CryptoCurrency:EC2/BitcoinTool.B!DNS` | An EC2 instance is mining cryptocurrency — it is compromised |
| Medium | `UnauthorizedAccess:IAMUser/MaliciousIPCaller` | API calls from a known bad IP — credentials may be stolen |
| Low | `Recon:EC2/PortProbeUnprotectedPort` | Someone is scanning your ports — could be routine internet noise |

> **💡 In a real SOC (Security Operations Center):** High-severity findings trigger immediate incident response. Medium findings are investigated within hours. Low findings are reviewed in daily or weekly security reviews.

## Troubleshooting

| Issue | What It Means | How to Fix It |
|-------|--------------|---------------|
| `A]detector already exists for the current account` | GuardDuty is already enabled in this region | Run `aws guardduty list-detectors --region us-east-1` to get the existing detector ID and use that |
| `create-sample-findings` returns an error | The detector ID may be wrong | Double-check the detector ID from Step 4 or Step 3 |
| `list-findings` returns empty `FindingIds` | Sample findings may take a moment to generate | Wait 30 seconds and try again |
| `get-findings` returns an error about finding IDs | The finding ID format is wrong | Make sure you copied the full finding ID (it is a long hex string with no dashes) |
| Console shows no findings | You may be in the wrong region | Check that the console is set to **US East (N. Virginia) us-east-1** in the top-right corner |
